This page is the technical architecture reference for the [demo repo](https://github.com/jajera/event-driven-serverless-platform-demo). For the end-to-end data flow and walkthrough path, start with [Project Overview](/aws-event-driven-serverless-walkthrough/introduction/project-overview/). For per-component runtime behavior, see [Infrastructure Layers](/aws-event-driven-serverless-walkthrough/architecture/infrastructure-layers/).

The platform is a single monorepo provisioned entirely through <Tooltip term="terraform" label="Terraform" />. Five modules compose the stack; each owns shared resources in its layer rather than duplicating infrastructure definitions.

## Staged apply order

Processing must exist before automated ingest starts — otherwise raw <Tooltip term="rinex" label="RINEX" /> files land in <Tooltip term="s3" label="S3" /> with no consumer.

Recommended sequence:

1. **Processing** — SQS, processor Lambda, DynamoDB Jobs table (creates ingest bucket dependencies)
2. **Ingest scheduler** — <Tooltip term="eventbridge-scheduler" label="EventBridge Scheduler" /> once processing is healthy
3. **Full apply** — presentation, observability, and reconciliation

Commands and Amplify options are in [Staged Apply](/aws-event-driven-serverless-walkthrough/deploy/staged-apply/).

## Key design decisions

### Provisioning and compute

| Decision                      | Rationale                                                     |
| ----------------------------- | ------------------------------------------------------------- |
| Terraform-only IaC            | Single workflow; no drift between tools                       |
| Processor as Lambda container | SQS-triggered container image — see design notes below        |
| External container image      | Built independently; mirrored to ECR at deploy time           |
| Python 3.14 zip Lambdas       | Ingest, query, and reprocess use managed runtime `python3.14` |

### Eventing and queues

| Decision                                                               | Rationale                                                      |
| ---------------------------------------------------------------------- | -------------------------------------------------------------- |
| <Tooltip term="eventbridge-scheduler" label="EventBridge Scheduler" /> | Recurring UTC triggers with dedicated invoke roles             |
| SQS standard queues                                                    | Processing order irrelevant; higher throughput than FIFO       |
| Dual queues (ingest vs reprocess)                                      | Separate concurrency caps — operator jobs do not starve ingest |
| Visibility timeout 900s                                                | Covers processor execution and SQS retry window                |

### Data layout and API

| Decision                                                  | Rationale                                                                                        |
| --------------------------------------------------------- | ------------------------------------------------------------------------------------------------ |
| <Tooltip term="parquet" label="Parquet" /> primary output | Efficient range scans; JSON fallback when `pyarrow` unavailable                                  |
| Deterministic output keys                                 | Idempotent reprocessing via safe overwrites                                                      |
| Prefix-based catalog                                      | Stations and dates from `processed/tec/station=.../year=.../doy=...` layout — not S3 Annotations |
| <Tooltip term="dynamodb" label="DynamoDB" /> for jobs     | Low-latency `job_id` lookups; no joins required                                                  |
| NaN sanitization in Query API                             | Non-finite floats become JSON `null` for browser compatibility                                   |

<Aside type="note">
  **Also considered — AWS Batch on Fargate:** An earlier design used Batch to
  scale **parallel calibration workers** as ingest volume grew. A limit-increase
  request for higher Batch/Fargate parallel execution was **rejected**, so the
  same container image (<code>ghcr.io/platformfuzz/tec-processor-image</code>)
  runs as an SQS-triggered{" "}
  <Tooltip term="processor-lambda" label="Lambda container" /> instead — each
  invocation pulls one message from{" "}
  <Tooltip term="process-queue" label="Process_Queue" /> or{" "}
  <Tooltip term="reprocess-queue" label="Reprocess_Queue" />. Parallelism comes
  from SQS event-source concurrency, not a Batch compute environment. Trade-off:
  900s Lambda timeout and Lambda concurrency caps vs Batch worker scaling.
</Aside>

<Aside type="note">
  **Also removed — S3 Annotations:** S3 Annotations and annotations-indexer were
  explored for catalog discovery. The current deployment uses **prefix-based
  listing** on processed output keys under <code>processed/tec/</code>.
</Aside>

## Observability

All Lambdas emit structured JSON logs with <Tooltip term="trace-id" label="trace_id" />, `duration_ms`, `outcome`, and error fields on failure. The `observability` module provisions alarms (DLQ depth, processor health, queue backlog, ingest-sync errors), an SNS topic, and the **`event-driven-platform`** dashboard.

Alarm names, subscription steps, and dashboard verification are in [Alarms and Dashboard](/aws-event-driven-serverless-walkthrough/verification/alarms-dashboard/).
