The <Tooltip term="jobs-table" label="Jobs table" /> tracks reprocessing jobs submitted through the <Tooltip term="reprocess-api" label="Reprocess API" />. The <Tooltip term="processor-lambda" label="processor Lambda" /> updates status when a `job_id` is present in the <Tooltip term="sqs" label="SQS" /> message.

## Table design

- **Partition key:** `job_id` (String) — no sort key
- **Provisioned by:** `processing` <Tooltip term="terraform" label="Terraform" /> module

## Status lifecycle

```text
queued → processing → completed
                    → failed
```

<table>
  <thead>
    <tr>
      <th>Status</th>
      <th>Meaning</th>
    </tr>
  </thead>
  <tbody>
    <tr>
      <td>
        <strong>queued</strong>
      </td>
      <td>Job record created, SQS message enqueued</td>
    </tr>
    <tr>
      <td>
        <strong>processing</strong>
      </td>
      <td>Processor Lambda has started the message</td>
    </tr>
    <tr>
      <td>
        <strong>completed</strong>
      </td>
      <td>
        Calibration succeeded, <code>output_key</code> populated
      </td>
    </tr>
    <tr>
      <td>
        <strong>failed</strong>
      </td>
      <td>
        Processing failed after retries or <Tooltip term="dlq" label="DLQ" />{" "}
        routing
      </td>
    </tr>
  </tbody>
</table>

## Lookup from Terraform

```bash
terraform -chdir=terraform output -raw jobs_table_name
```

## Poll via API

```bash
export API_URL="$(terraform -chdir=terraform output -raw api_url)"
curl "${API_URL}/reprocess/${JOB_ID}"
```

See [API Response Schemas](/aws-event-driven-serverless-walkthrough/data-contract/api-response-schemas/) for the GET response format.
